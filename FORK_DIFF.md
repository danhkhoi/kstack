# kstack Fork Differences

This document tracks all intentional divergences between **kstack** (this repo,
`danhkhoi/kstack`) and the upstream **gstack** (`garrytan/gstack`). Use it to
understand what changed, why, and what to watch for when the upstream releases
new versions.

---

## How Gary's Setup + Telemetry System Works

Understanding the upstream design helps you know what we removed and why.

### Setup script (`setup`)

`setup` is the entry point for installing gstack. It:

1. Builds the `browse` binary (Playwright headless Chromium CLI) via `bun build`
2. Ensures Playwright's Chromium is installed (`bunx playwright install chromium`)
3. Creates `~/.gstack/projects/` state directory
4. Links sub-skill directories into `~/.claude/skills/` so each skill becomes a
   slash command (e.g. `~/.claude/skills/qa` → `gstack/qa` → `/qa`)
5. Shows a welcome message on first run

### Telemetry system (what we removed)

Gary's gstack has an opt-in telemetry system to collect usage data into Supabase.
It works in three layers:

**Layer 1 — Preamble bash (runs at the start of every skill):**
Every SKILL.md has a `## Preamble (run first)` block generated by
`scripts/gen-skill-docs.ts`. In the upstream, this preamble:
- Reads `~/.gstack/config.yaml` for the telemetry tier (`off` / `anonymous` / `community`)
- Checks `~/.gstack/.telemetry-prompted` to see if the opt-in prompt was shown
- Writes a basic event to `~/.gstack/analytics/skill-usage.jsonl`
- Creates a `.pending-{session-id}` marker file (used to catch interrupted sessions)

**Layer 2 — Telemetry prompt (first-run only):**
On first use, the skill preamble asks the user via `AskUserQuestion`:
- *Community mode*: shares which skills you use + stable device ID
- *Anonymous mode*: just a counter (no ID)
- *Off*: nothing sent

The tier is saved via `bin/gstack-config set telemetry <tier>`.

**Layer 3 — Sync to Supabase (`bin/gstack-telemetry-sync`):**
After each skill run, `bin/gstack-telemetry-log` appends a structured JSON event
to `~/.gstack/analytics/skill-usage.jsonl` and then calls `gstack-telemetry-sync`
in the background. The sync script reads unsynced lines from the JSONL, strips
local-only fields (`_repo_slug`, `_branch`), and POSTs them to a Supabase
endpoint (configured via `supabase/config.sh`). Rate-limited to once per 5 minutes.

**Layer 4 — Update check pings Supabase:**
`bin/gstack-update-check` also fires a background `curl` to Supabase's
`/update_checks` table when it fetches the remote version, recording install
counts by OS/version.

### `bin/gstack-config`

A small bash utility that reads/writes `~/.gstack/config.yaml`. Used by skills to
store: `telemetry`, `proactive`, `auto_upgrade`, `update_check`, `gstack_contributor`.

### `bin/gstack-update-check`

Checks the remote `VERSION` file on GitHub and compares against local.
Outputs `UPGRADE_AVAILABLE <old> <new>` or `JUST_UPGRADED <old> <new>`.
Caches results (60 min for up-to-date, 720 min for upgrade-available).
Supports snooze with escalating backoff: 24h → 48h → 7 days.

### Skill name convention

In upstream gstack:
- The root `SKILL.md` at `~/.claude/skills/gstack/SKILL.md` → `/gstack`
- Sub-skills are linked **without prefix**: `~/.claude/skills/qa` → `gstack/qa` → `/qa`
- `gstack-upgrade/SKILL.md` → `~/.claude/skills/gstack-upgrade` → `/gstack-upgrade`

## What kstack Changed

### 1. Telemetry completely removed

| File | Change |
|------|--------|
| `bin/gstack-telemetry-log` | No-op stub — cleans pending markers, exits 0 |
| `bin/gstack-telemetry-sync` | No-op stub — exits 0 immediately |
| `scripts/gen-skill-docs.ts` | Removed telemetry vars from preamble bash (`_TEL`, `_TEL_PROMPTED`, `_TEL_START`, `_SESSION_ID`, JSONL write, `.pending-*` loop) |
| `scripts/gen-skill-docs.ts` | Removed `generateTelemetryPrompt()` from `generatePreamble()` — no more opt-in prompt |
| `scripts/gen-skill-docs.ts` | Removed "Telemetry (run last)" epilogue from `generateCompletionStatus()` |
| `bin/gstack-update-check` | Removed Supabase install ping from the slow-path fetch |
| `bin/gstack-update-check` | Removed `gstack-telemetry-log` call for `upgrade_prompted` events |

**No data is sent anywhere.** Skill usage, durations, outcomes, version pings — none of it.

### 2. Rebranded: gstack → kstack

| What | Upstream (gstack) | This fork (kstack) |
|------|-------------------|---------------------|
| State directory | `~/.gstack/` | `~/.kstack/` |
| Config file | `~/.gstack/config.yaml` | `~/.kstack/config.yaml` |
| Config bin | `bin/gstack-config` | `bin/kstack-config` |
| Update-check bin | `bin/gstack-update-check` | `bin/kstack-update-check` |
| Skill install root | `~/.claude/skills/gstack/` | `~/.claude/skills/kstack/` |
| Root skill command | `/gstack` | `/kstack` |
| Env var prefix | `GSTACK_*` | `KSTACK_*` |
| Remote VERSION URL | `garrytan/gstack` on GitHub | `danhkhoi/kstack` on GitHub |

### 3. Sub-skills are prefixed to avoid conflict

**Problem:** If both `garrytan/gstack` and `danhkhoi/kstack` are installed, they'd
both try to create `~/.claude/skills/qa`, `~/.claude/skills/review`, etc., and
overwrite each other.

**Fix:** `setup`'s `link_claude_skill_dirs()` now auto-prefixes sub-skill symlinks
with the install directory basename:

```
~/.claude/skills/kstack-qa       → kstack/qa       → /kstack-qa
~/.claude/skills/kstack-review   → kstack/review   → /kstack-review
~/.claude/skills/kstack-ship     → kstack/ship     → /kstack-ship
# etc.
```

This means all skill commands have the `kstack-` prefix. No overlap with gstack's
unprefixed commands.

### 4. `gstack-upgrade/` renamed to `upgrade/`

Source directory renamed `gstack-upgrade/` → `upgrade/` so that the auto-prefix
logic produces `kstack-upgrade` (not `kstack-gstack-upgrade`).
All references updated: paths, state dir, clone URL.

## Staying Independent from Upstream

### What to watch when Gary releases a new gstack version

Check Gary's [CHANGELOG](https://github.com/garrytan/gstack/blob/main/CHANGELOG.md)
and compare against our changes. The main risk areas:

| Area | Risk | What to do |
|------|------|------------|
| `scripts/gen-skill-docs.ts` | Upstream adds new preamble features (new bash vars, new sections) | Merge selectively — skip anything that reintroduces telemetry or `~/.gstack` paths |
| New skill templates (`*/SKILL.md.tmpl`) | New skills added upstream | Copy the `.tmpl` file; it will automatically get kstack paths when regenerated |
| `setup` script | Upstream changes skill linking logic | Re-apply our auto-prefix patch on top of upstream changes |
| `bin/gstack-update-check` | Upstream adds new Supabase pings | Use our `bin/kstack-update-check` instead — do NOT copy upstream's version directly |
| `bin/gstack-config` | Upstream adds new config keys | Copy new features into `bin/kstack-config`, keeping `~/.kstack` state dir |
| `browse/` source | Browse CLI changes | Safe to merge — no telemetry lives here |

### Merging upstream changes

```bash
# Add upstream remote (one time)
git remote add upstream https://github.com/garrytan/gstack.git

# Fetch latest
git fetch upstream

# See what changed
git log upstream/main..HEAD --oneline          # our commits
git log HEAD..upstream/main --oneline          # upstream commits we don't have

# Cherry-pick safe changes (browse/, new skill templates, bug fixes)
git cherry-pick <commit-sha>

# After merging template changes, always regenerate
bun run gen:skill-docs
```

**Never merge directly:** `bin/gstack-update-check`, `bin/gstack-telemetry-*`,
or upstream changes to `generatePreambleBash` / `generateCompletionStatus` in
`gen-skill-docs.ts` without reviewing for new telemetry calls first.
